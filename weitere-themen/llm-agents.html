01 — Tokenisierung: Alles ist Text

Ein LLM verarbeitet keine Zeichen, sondern Token-IDs — Ganzzahlen aus einem festen Vokabular. Byte-Pair Encoding (BPE) baut dieses Vokabular bottom-up auf: häufige Zeichenpaare werden iterativ zusammengefasst.

GPT-2 Vokabular: ~50 000 BPE-Tokens
Token-Count Beispieltext: 70
GPT-4o Vokabular: ~200 000 BPE-Tokens
Token-Count Beispieltext: 54 (effizienter)

Code ist nur weiterer Text. Ein HTML-Snippet wie <div id="main"> wird genauso tokenisiert wie natürliche Sprache — der Transformer unterscheidet nicht zwischen beiden. Bilder werden durch einen visuellen Encoder in eine Sequenz von Bild-Tokens übersetzt, bevor sie an das LLM übergeben werden (z.B. "The cat sat on the mat" → 464, 3797, 3332, ...).

02 — Der LLM-Training-Stack

Das Training moderner LLMs schichtet drei Lernparadigmen übereinander — der Foliensatz beschreibt es als "Kuchen": das Unsupervised Pre-Training ist die Masse, Supervised Fine-Tuning ist die Glasur, Reinforcement Learning ist die Kirsche.

Reinforcement Learning (die Kirsche) Human Feedback (RLHF) oder verifizierbare Rewards (RLVR) — verfeinert das Verhalten
Supervised Fine-Tuning (die Glasur) Lernen aus annotierten Beispielen — Chat-Format, Instruktions-Folgsamkeit
Unsupervised Pre-Training (die Masse) Next-Token-Prediction auf dem Web — die große Datenmenge

\[ \mathcal{L}(\theta) = -\sum_k \log p_\theta(x_k \mid x_{k-1}, \ldots, x_0) \quad \text{(maximum likelihood)} \]
02
Äquivalenz zur Cross-Entropie
\[ \mathcal{L}(\theta) = \sum_k \mathrm{CE}(\mathrm{logits}_\theta,\; x_k) \]
Next-Token-Prediction ist ein Klassifikationsproblem: das Netz wählt aus dem Vokabular das wahrscheinlichste nächste Token. Dieser Verlust gilt für Pre-Training und SFT gleichermaßen — beim Chat-Training werden nur die Antwort-Token zum Verlust beigetragen.

Supervised Fine-Tuning (SFT) für einen Chat-Bot: Die Eingabe enthält Spezial-Tokens wie <|im_start|>system, <|im_start|>user, <|im_start|>assistant. Der Verlust wird nur über die Antwort-Token des Assistenten berechnet.

Kaplan 2020 Power Law

Die Leistung von LLMs folgt Potenzgesetzen in Bezug auf Compute, Datensatzgröße und Parameteranzahl — und zwar unabhängig voneinander, solange der jeweils andere Faktor nicht zur Engstelle wird.

05 — Reinforcement Learning mit verifizierbaren Rewards (RLVR)

RLVR Policy Gradient GRPO

Bei RLVR gibt ein Verifizierer (kein Reward-Modell) den Reward: er prüft maschinell, ob die Antwort korrekt ist.

\[ r(x_{1:T}) = \begin{cases} 1 & \text{wenn die Antwort korrekt ist} \\ 0 & \text{sonst} \end{cases} \]

Maximiertes Ziel:

\[ \max_\theta\; \mathbb{E}_{x_{1:T} \sim p_\theta}\!\bigl[r(x_{1:T})\bigr] \]
01
Policy-Gradient-Theorem
\[ \nabla_\theta\, \mathbb{E}[r] = \mathbb{E}\!\left[\bigl(r(x_{1:T}) - b\bigr)\,\nabla_\theta \log p_\theta(x_{1:T})\right] \]
Das Subtrahieren einer beliebigen Konstanten b lässt den erwarteten Gradienten unverändert. Man wählt b = r̄ (mittlerer Reward über den Batch) — damit entsteht eine reward-gewichtete Cross-Entropie:
\[ = -\mathbb{E}\!\left[\bigl(r(x_{1:T}) - b\bigr)\,\nabla_\theta \sum_t \mathrm{CE}(\mathrm{logits}_\theta,\, x_t)\right] \]
02
GRPO — Group-Relative Policy Optimization Für jede Frage q werden G Antworten o₁, …, o_G gesampelt und bewertet. Die Baseline ist der Gruppen-Mittelwert (kein Wert-Netzwerk nötig):
\[ A_i = \frac{r_i - \bar{r}}{\mathrm{std}(\mathbf{r})}, \qquad \bar{r} = \frac{1}{G}\sum_{i=1}^{G} r_i \]
Quelle: DeepSeekMath — Shao et al. 2024.

Reasoning Trace: Durch RLVR lernen Modelle, in einem <thinking>-Block zu denken, Fehler zu korrigieren und erst dann die finale Antwort (z.B. #### 72) zu geben.

06 — DPO: Direct Preference Optimization

RLHF-Alternative Bradley–Terry Rafailov 2023

DPO trainiert das LLM direkt auf menschlichen Präferenzen, ohne ein separates Reward-Modell zu benötigen. Wenn ein Mensch Antwort A gegenüber B bevorzugt (A ≻ B):

Prompt
→ p_θ →
Output A
Output B
A ≻ B
01
Einfache DPO-Loss (Bradley–Terry)
\[ \mathcal{L}(\theta) = -\log \sigma\!\bigl(\log p_\theta(A) - \log p_\theta(B)\bigr) \]
02
Logistische Version — mit Referenzmodell
\[ \mathcal{L}(\theta) = -\log \sigma\!\left(\beta\!\left[ \log \frac{p_\theta(A)}{p_\text{ref}(A)} - \log \frac{p_\theta(B)}{p_\text{ref}(B)} \right]\right) \]
Der Referenzmodell-Term \(p_\text{ref}\) verhindert, dass das Modell zu weit vom ursprünglichen Pre-Training-Verhalten abweicht. β steuert die Stärke dieser KL-Regularisierung.

07 — Coding Agents: Function Calling & Tool-Use

Function Calling Tool-Use

Ein LLM kann den Token-Generierungsloop verlassen, um ein externes Tool aufzurufen. Das Modell erzeugt einen strukturierten Aufruf, das Tool läuft, das Ergebnis wird in den Kontext eingefügt und die Generierung geht weiter.

<python>
import math
print(48 + 24)
</python>

<results>
72
</results>
Beim </python>-Tag: 1. Token-Generierung pausieren
2. Code mit Python ausführen
3. stdout in <results> kopieren
4. Generierung fortsetzen

Ein minimaler Coding Agent braucht nur wenige Werkzeuge:

LLM
function call → ← result
read_file
write_file
bash
grep / search

Mit diesen vier Werkzeugen kann ein LLM vollständige Coding-Aufgaben lösen: Code lesen, schreiben, ausführen und im Codebase suchen. Das ist das Prinzip hinter Tools wie Claude Code.

08 — MCP: Model Context Protocol (Anthropic)

JSON-RPC 2.0 Anthropic Standardprotokoll

MCP standardisiert die Kommunikation zwischen LLM-Host, Client und externen Tool-Servern über JSON-RPC 2.0. Das LLM generiert eine JSON-Anfrage, der Server verarbeitet sie und gibt das Ergebnis zurück.

LLM generiert die Anfrage:

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "ping"
}

… zurückgegeben vom MCP-Server:

{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {}
}

MCP ermöglicht es, LLM-Clients (Claude Desktop, IDE-Extensions) mit beliebigen Tool-Servern zu verbinden — ohne Custom-Integration pro Tool. Server registrieren ihre verfügbaren Methoden beim Sitzungsstart.
