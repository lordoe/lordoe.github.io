Kreuzvalidierung

Um zu wissen, wie gut ein Modell auf unbekannten Daten funktioniert, braucht man ein systematisches Evaluationsverfahren. Cross-Validation löst das Problem des unstabilen Train/Test-Splits.

Das Evaluationsproblem — warum drei Mengen?

Wirklich unbekannte Daten haben keine Labels. Die Lösung: Einen Teil der gelabelten Daten vom Training ausschließen und damit den Einsatz auf unbekannten Daten simulieren (Holdout-Methode). Sobald man jedoch die Hyperparameter (z. B. Polynomgrad, k bei k-NN) ebenfalls anhand der Testmenge optimiert, verliert sie ihre Unabhängigkeit — daher braucht man eine dritte Menge.

Training Set

Das Modell wird auf diesen Daten trainiert. Typisch: 60–70 % der Gesamtdaten.

Validation Set

Zum Tuning von Hyperparametern. 10–20 %. Wird während der Modellauswahl genutzt.

Test Set

Einmalig am Ende — liefert eine unverzerrte Schätzung der Generalisierungsfähigkeit.

Training ~65 %
Val ~15 %
Test ~20 %

Typische Aufteilung: 80/20 oder 66/33 für Training/Test. Ergebnisse variieren stark je nach Wie der Split gemacht wird.

Formeln — aus der Vorlesung
k-fold CV — Gesamtschätzung (Mittelwert) \[ E = \frac{1}{k} \sum_{i=1}^{k} E_i \]

\(E_i\) = Performance auf dem \(i\)-ten Validation-Fold; \(k\) = Anzahl Folds

Standardabweichung über Folds \[ \sigma = \sqrt{\frac{1}{k} \sum_{i=1}^{k} (E_i - E)^2} \]

Niedrigere Std.-Abweichung = stabilere Ergebnisse. Beim Vergleich zweier Klassifikatoren immer auch \(\sigma\) berücksichtigen.

Leave-One-Out CV (LOO-CV) — Spezialfall k = n \[ E_{\text{LOO}} = \frac{1}{n} \sum_{i=1}^{n} E_i \]

Jeder der \(n\) Datenpunkte wird genau einmal als Validation-Datenpunkt verwendet. Maximale Nutzung der Trainingsdaten, aber hoher Rechenaufwand.

Accuracy — Evaluationsmaß \[ \text{Accuracy} = \frac{TP + TN}{TP + FP + TN + FN} = \frac{TP + TN}{\text{\# Samples}} \]
Bias-Variance-Tradeoff bei der Wahl von k

Die Wahl von k beeinflusst, wie viele Daten zum Training zur Verfügung stehen und wie stabil die Schätzung ist. In der Praxis wird oft k = 10 verwendet. Bei kleinen Datensätzen: kleineres k (nicht zu kleine Trainingsmengen). Bei sehr großen Datensätzen: ebenfalls kleineres k (aus Laufzeitgründen). Läufe können parallelisiert werden.

Typische Fehler bei der Evaluation
Data Leakage: Preprocessing (z. B. Normalisierung, Feature-Selektion) auf dem gesamten Datensatz vor dem Split durchführen — statt nur auf den Trainingsdaten. Dadurch "sieht" das Modell indirekt Validierungsdaten. Ergebnis: zu optimistische Fehlerrate.
Test-Set-Kontamination: Das Test-Set zum Hyperparameter-Tuning nutzen. Es muss strikt einmalig am Ende verwendet werden.
Falsches Splitting bei Zeitreihen: Bei zeitabhängigen Daten darf das Validation-Set nie Zeitpunkte vor dem Training-Set enthalten. Zufälliges Shufflen führt zu Data Leakage aus der Zukunft.
Training-Fehler als Evaluationsmaß: Vollständig aufgebaute Decision Trees — 100 % korrekt auf Trainingsdaten. Perceptron auf linear separierbaren Daten: ebenfalls 100 %. Kein Hinweis auf Generalisierungsfähigkeit.
Vergleich — Holdout vs. Cross-Validation
Holdout-Methode

Einfach, schnell. Ergebnisse variieren stark je nach zufälligem Split. Wenig Daten zum Training bei kleinen Datensätzen.

k-fold CV

Stabilere Schätzung. Jeder Datenpunkt wird genau einmal zur Validation verwendet. Laufzeit steigt linear mit k — aber parallelisierbar.

LOO-CV

Maximale Trainingsdata-Nutzung. Bei \(n\) Datenpunkten: \(n\) Modelltrainings. Praktisch nur für kleine Datensätze.
